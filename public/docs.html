Getting Started with ObsAI

ObsAI is an agentic AI assistant for Splunk, Cribl, and Observability administration. It uses Retrieval-Augmented Generation (RAG) with 7 specialized knowledge collections, 133 skills, and 54 agent personas to provide accurate, grounded answers about your environment.

Quick Start

Start all 8 containers with a single command:

# Clone and start
git clone <repo-url> && cd chainlit
cp .env.example .env          # Edit credentials
bash docker_files/start_all.sh

# Open in browser
http://localhost:8000

Just type naturally. ObsAI auto-detects your intent and routes to the best specialist:

How do I use the stats command?
Optimize: index=main | stats count by host | sort -count
Show my saved searches in org-search

Write an mstats query for CPU utilization by host

Auto-Detection

When using the General profile, ObsAI auto-detects your intent using keyword analysis:

  • Possessive patterns ("my inputs.conf", "our saved searches") → Org Expert
  • App references ("org-search", "TA-nmap") → Org Expert
  • Error keywords ("not working", "debug", "fix") → Troubleshooter
  • SPL keywords ("optimize query", "stats command", "tstats") → SPL Expert
  • Cribl keywords ("pipeline", "route", "Cribl Stream") → Cribl Expert
  • Metrics keywords ("mstats", "OpenTelemetry", "SLI") → Observability
Dynamic profile tips: If ObsAI detects your query would be better served by a different profile, it suggests switching. For example, asking a .conf question while on the SPL Expert profile will suggest switching to Config Helper.

Context Ingestion

Add context to any message using ingestion directives. The content is chunked, embedded, and stored in the vector store for immediate retrieval. Supports URLs, local files, and inline text.

Directives

read_url

Fetch and ingest content from a URL:

read_url: https://docs.splunk.com/Documentation/Splunk/latest/SearchReference/Stats
What does the stats command do?

read_file

Ingest a local file (accessible from the container):

read_file: /app/documents/my_custom.conf
Show me the stanzas in this file

read_text

Ingest inline text directly:

read_text: "index=main sourcetype=syslog | stats count by host"
Explain this SPL query

bash ingest_specs/run_ingest_all.sh

# Org configs + local docs only

Feedback System

Your feedback directly improves ObsAI's future responses through a closed-loop learning system. Every thumbs up or down trains the assistant to give better answers.

How It Works

Thumbs Up

The Q&A pair is polished by the LLM into a clean canonical example, then stored in the feedback_qa collection with 25x retrieval weight. Future queries matching this topic will preferentially retrieve the validated answer.

# What happens internally:
1. User clicks thumbs up
2. LLM rewrites Q&A into clean canonical form
3. Stored in feedback_qa collection (25x boost)
4. Search optimizer indexes new pattern
5. Future similar queries retrieve this answer first

Thumbs Down

You'll be prompted for a reason. The LLM creates a "what NOT to do" warning example, stored in negative_feedback and injected as guardrails into future prompts to prevent similar mistakes.

Learning Cycle

All feedback triggers the search optimizer's learning cycle to extract optimization patterns. The self-adaptive RAG adjusts collection weights based on which sources produced good vs. bad answers.

Impact: A single thumbs up can permanently improve responses for an entire topic area. Feedback is the fastest way to customize ObsAI for your environment.

How It Works

Workflows are sequences of nodes that execute in dependency order. Each node represents an action — invoking a skill, dispatching to an agent, evaluating a condition, or accepting/producing data. When you execute a workflow, the backend runs each node in sequence (or parallel when dependencies allow), passing context between steps.

User clicks "Execute" on a workflow template
  → Frontend sends node list to POST /workflows/execute
    → Backend converts nodes to WorkflowTasks
      → WorkflowOrchestrator executes in dependency order
        → Each Skill node calls skill_executor.py
        → Each Agent node calls agent_dispatcher.py
        → Condition nodes evaluate expressions
      → Results passed downstream between tasks
    → Final WorkflowResult returned with metrics
  → Frontend displays execution results

langfuse:

  secret_key: sk-obsai-dev  # Change in production

When enabled, the chat_app/langfuse_integration.py module automatically instruments all LLM calls made through llm_utils.py. Every chat response generates a Langfuse trace with nested spans for:

  • Intent classification
  • Vector store retrieval (each collection queried)
  • Knowledge graph context generation
  • LLM chat completion (main response)
  • Confidence scoring

Authentication & Security

ObsAI supports configurable authentication with role-based access control. Authentication is disabled by default for development and can be enabled for production deployments.

Authentication Flow

Authentication flow: Login credentials produce a JWT cookie, validated on each request with 3-tier resolution

Enabling Authentication

Set the environment variable before starting containers:

# In .env file or docker-compose.yml
ENABLE_AUTHENTICATION=true

# Default admin credentials (CHANGE IN PRODUCTION)
ADMIN_USER=admin
ADMIN_PASSWORD=admin

When enabled, users see a login screen. When disabled (default for development), anonymous access is granted with full admin privileges.

When authentication is disabled, all admin endpoints are accessible without restriction (intended for development only).

Security Best Practices

Production checklist:
  • Change default admin password: ADMIN_PASSWORD=your_secure_password
  • Generate auth secret: CHAINLIT_AUTH_SECRET=$(openssl rand -hex 32)
  • Generate JWT secret: JWT_SECRET=$(openssl rand -hex 32)
  • Change PostgreSQL credentials from defaults
  • Enable authentication: ENABLE_AUTHENTICATION=true
  • Enable HTTPS: configure SSL/TLS with a trusted certificate (see SSL/TLS section below)
  • Restrict network access to admin ports (5432, 8001, 9090)

Session Management

Chainlit manages authentication sessions with JWT tokens stored in browser cookies:

  • Session timeout: 1 hour of inactivity
  • User session expiry: 15 days
  • Session history: Available when authentication is enabled (chat threads persist across sessions)

SSL/TLS (HTTPS)

ObsAI supports HTTPS via SSL/TLS certificates. When enabled, the application serves all traffic over HTTPS using uvicorn's built-in SSL support.

Using a Reverse Proxy (Recommended for Production)

For production deployments, consider using a reverse proxy (Nginx, Caddy, Traefik) in front of ObsAI for SSL termination, which provides better performance, certificate auto-renewal (Let's Encrypt), and HTTP/2 support:

# Example: Nginx reverse proxy with SSL termination
server {
    listen 443 ssl;
    server_name obsai.example.com;

    ssl_certificate     /etc/ssl/certs/obsai.crt;
    ssl_certificate_key /etc/ssl/private/obsai.key;

    location / {
        proxy_pass http://localhost:8000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

Orchestration Framework

The orchestration framework provides 22 strategies for coordinating 54 agent personas, 133 skills, and reasoning patterns. It dynamically selects the best strategy based on query complexity, resource availability, and intent classification. Strategies range from simple single-agent to complex multi-agent governance models.

Orchestration Flow

Every user query flows through this pipeline before a response is generated:

1

Intent Classification

Classify user query intent: SPL generation, config help, analysis, greeting, etc. Determines which skills and agents are relevant.

2

Strategy Selection

Check per-intent overrides → adaptive picks best strategy → resource fallback if constrained. Heavy strategies auto-downgrade.

3

Agent Dispatch

Route to best-fit agent persona from 54 agents across 10 departments based on intent + keyword matching.

4

Skill Execution

4-tier handler resolution: Tool Registry + MCP → Skills Manager → Internal Handlers → ReAct Loop fallback.

5

Quality Check

Critic agent reviews output quality (adaptive/review_critique strategies). Iterates if below quality threshold.

6

Response

Synthesize final response with sources, confidence score, and reasoning trace. Stream to user if enabled.

Resource Management

Strategies are automatically downgraded when system resources are constrained. This prevents resource exhaustion and ensures the system remains responsive:

  • Heavy → Medium: When can_run_heavy_task() returns false (CPU > 80% or memory > 85%)

  • Fallback is transparent to the user — the system automatically picks the next-best strategy
  • Override mappings configurable via config.yaml orchestration.strategy_overrides
Per-intent overrides: You can force specific strategies for specific intents. For example, SPL generation uses single_agent for speed (no multi-agent overhead), while security audits might use parallel for thorough analysis. Configure in orchestration.strategy_overrides.

The Knowledge Graph enhances the RAG pipeline by injecting structural facts into the LLM context. When a user asks about an SPL command, the graph contributes facts that vector similarity alone would miss:

kg = get_knowledge_graph()

MCP Gateway

The Model Context Protocol (MCP) Gateway enables the assistant to discover and invoke external tools dynamically. MCP provides a standardized protocol for connecting AI systems with data sources, APIs, and services — turning any compatible server into an extension of the assistant's capabilities.

What is MCP?

MCP (Model Context Protocol) is an open standard for AI-tool communication. Instead of hardcoding tool integrations, MCP lets tools describe themselves (capabilities, parameters, authentication) and lets AI systems discover and use them at runtime.

1

Dynamic Discovery

Tools are discovered at runtime. Add a new MCP server and its capabilities are instantly available without code changes or restarts.

2

Standardized Interface

All tools speak the same protocol. Whether it's Splunk, GitHub, or SharePoint, the interface is consistent — JSON Schema parameters and structured responses.

3

Auth Isolation

Each server handles its own authentication. Bearer tokens, API keys, and OAuth are managed independently per server.

4

Transport Flexibility

Connect via SSE (real-time), Streamable HTTP (cloud), or Stdio (local CLI) — one gateway handles all transport types.

Transport Types

The gateway supports three transport protocols for connecting to MCP servers:

TransportProtocolBest ForExample
SSEServer-Sent Events over HTTPLocal MCP servers, real-time tool callsSplunk MCP at http://127.0.0.1:8181
Streamable HTTPHTTP request/response with streamingCloud-hosted MCP servers, remote APIsGitHub MCP at https://github.example.com/mcp
StdioStandard I/O pipes (stdin/stdout)CLI-based tools, local scriptsSharePoint MCP via uvx sharepoint-mcp

MCP in the Execution Pipeline

MCP tools integrate at Tier 1 of the 4-tier skill resolution chain, alongside the built-in Tool Registry:

# Execution flow:
# User Query
#   → Intent Classifier (classify intent)
#   → Orchestrator (pick strategy: adaptive, parallel, etc.)
#   → Agent Dispatcher (route to best-fit agent persona)
#   → Skill Executor (4-tier resolution)
#       Tier 1: Tool Registry + MCP Gateway ← MCP tools discovered here
#       Tier 2: Skills Manager (Python packages)
#       Tier 3: Internal Handlers (57 built-in)
#       Tier 4: ReAct Loop (LLM fallback)
#   → Response with sources and confidence

MCP + Orchestration

MCP tools integrate seamlessly with all 17 orchestration strategies. The orchestrator can route MCP calls through any strategy:

  • Single Agent — One agent calls MCP tools directly for simple queries
  • Parallel — Multiple agents call different MCP servers simultaneously
  • Adaptive — Auto-selects best strategy and routes MCP calls accordingly
  • Voting — Multiple agents use MCP tools and vote on best result
  • Human-in-Loop — MCP tool calls wait for user approval before execution
  • Review/Critique — Primary agent calls MCP tools, critic reviews results

Building a Custom MCP Server

# Python example using the MCP SDK
from mcp.server import Server, NotificationOptions
from mcp.types import Tool, TextContent

app = Server("my-custom-mcp")

@app.list_tools()
async def list_tools():
    return [
        Tool(
            name="search_logs",
            description="Search application logs",
            inputSchema={
                "type": "object",
                "properties": {
                    "query": {"type": "string", "description": "Search query"},
                    "time_range": {"type": "string", "default": "1h"}
                },
                "required": ["query"]
            }
        )
    ]

@app.call_tool()
async def call_tool(name: str, arguments: dict):
    if name == "search_logs":
        results = search_my_logs(arguments["query"], arguments.get("time_range", "1h"))
        return [TextContent(type="text", text=str(results))]

# Run with SSE transport
app.run(transport="sse", port=8282)

Security Best Practices

  • Never hardcode tokens — Use environment variables (SPLUNK_MCP_TOKEN, GITHUB_TOKEN)
  • Minimize permissions — Use tokens with minimal required scope/role
  • Disable unused servers — Reduces attack surface; toggle via Admin UI
  • Set connection timeout — Prevents hanging connections (default: 30s)
  • Monitor MCP logs — Check for unauthorized access attempts or unusual tool calls
  • Use approval gates — Skills that call MCP tools can be set to CONFIRM or REVIEW gates

Troubleshooting

Common issues and their solutions. For real-time diagnostics, use /health in chat or check the Admin Console dashboard.

!

Services Not Starting

Check docker ps -a and docker logs <name> for startup errors.

!

Empty Collections

Run bash ingest_specs/run_ingest_all.sh inside the container to populate.

!

Slow Responses

Switch to LLM_LITE, reduce Search Depth, disable reranking. Check /health.

!

Auth Issues

Verify ADMIN_USER/PASSWORD in .env, check PostgreSQL running, clear cookies.

Services not starting

docker ps -a                        # Check all containers
docker logs chat_ui_app             # View app logs
docker logs llm_api_service         # View Ollama logs
docker logs chat_db_app             # View PostgreSQL logs

Collections empty after ingestion

bash docker_files/check_chroma.sh   # Check collection counts
bash ingest_specs/run_ingest_all.sh # Re-run ingestion

Slow responses

Performance tips: Switch to LLM_LITE profile for faster responses. Reduce Search Depth in settings. Disable reranking if enabled. Check system resources via /health or GET /api/system/resources.

Authentication issues

  1. Check ENABLE_AUTHENTICATION=true in .env
  2. Verify ADMIN_USER and ADMIN_PASSWORD are set
  3. For database users, check PostgreSQL is running: docker logs chat_db_app
  4. Reset admin password by updating .env and restarting
  5. Clear browser cookies if session token is stale

PostgreSQL version mismatch

If you see database files are incompatible with server, the data volume was initialized with a different PostgreSQL version than the current container image. Update docker_files/Dockerfile.postgres to match the data version (check logs for exact version numbers).

Embedding errors

If you see "truncating to prevent embedding error", chunk sizes may exceed the embedding model's context limit. The default (250 tokens) is tuned for mxbai-embed-large (512-token context).

Admin console "Access Denied"

  1. Ensure you're logged in with an ADMIN role user
  2. If auth is disabled, admin should be accessible to everyone
  3. Check browser developer tools Network tab for 401/403 responses
  4. Try clearing cookies and logging in again

Ollama connection failures

IPv6 on WSL2: Podman rootless may expose ports only via IPv6 ([::1]). ObsAI auto-detects this with _probe_ollama_url(). If Ollama is unreachable, check with: curl http://[::1]:11430/api/version

Container network issues

# Verify the bridge network exists
docker network ls | grep chainlit_net

# Check container connectivity
docker exec chat_ui_app ping -c 1 llm_api_service

# Check for orphaned processes (podman)
pgrep -fa chainlit
